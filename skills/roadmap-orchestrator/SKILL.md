# Roadmap Orchestrator

## Жёсткий контракт оркестратора

**Оркестратор не пишет содержимое артефактов.** Спека, план, код и документация создаются только заспавненными агентами.

Два исключения, которые остаются в основной сессии:
1. **Диалог с пользователем** — уточняющие вопросы, сбор решений, подтверждение этапа
2. **Учёт состояния конвейера** — отметки в роадмапе (`- [x]`), статусные заметки, чек-лист дымового теста

Если пользователь просит «просто напиши спеку сам» — заспавни писателя и передай ему, не пиши инлайн.

---

## Шаг 1. Приём роадмапа

Конвейеру нужен **рабочий роадмап** — чек-лист этапов (`- [ ]`), который оркестратор ведёт и отмечает. Это «позвоночник» всего процесса. Но в проекте его может не быть — тогда роадмап собирается из того, что есть.

Различать две вещи:
- **Источник замысла** — свободная форма: roadmap-файл, но также `PLAN.md`/`TODO.md`/`BACKLOG.md`/`MILESTONES.md`, раздел «Roadmap/Этапы/Plan» в `README`, vision/concept-доки в `docs/`, либо идея в голове пользователя.
- **Рабочий роадмап** — чек-лист этапов, с которого стартует конвейер. Итог Шага 1 — всегда он.

Формат рабочего роадмапа:
```markdown
## Этапы проекта
- [ ] Этап 1: описание
- [x] Этап 2: описание (завершён)
- [ ] Этап 3: описание
```

### Лестница обнаружения

Идти сверху вниз, остановиться на первой сработавшей ветке.

1. **Готовый чек-лист этапов уже есть.** Проверить `ROADMAP.md` в корне, `docs/roadmap*.md` / `docs/ROADMAP.md`, либо любой `*.md` с разделом `## Этапы` или списком `- [ ]`. Найден → взять как есть, перейти к выбору этапа.

2. **Замысел есть, но без структуры.** Проверить `PLAN.md`, `TODO.md`, `BACKLOG.md`, `MILESTONES.md`, раздел плана/roadmap в `README`, concept/vision-доки в `docs/`. Найдено → показать пользователю, что нашёл, и **предложить собрать из этого рабочий роадмап** в `ROADMAP.md`. Подтверждение обязательно.

3. **Ничего не нашлось.** Провести короткий диалог (разрешено, исключение 1): цель проекта/итерации и крупные куски работы. Собрать из ответов минимальный роадмап.

### Кто собирает роадмап (ветки 2–3)

- **Тонкий чек-лист (≈5–10 строк)** — оркестратор пишет **сам**: это учёт состояния конвейера (исключение 2), а не содержимое артефакта. Детальная проработка всё равно живёт в Фазе A (спека) и Фазе B (план).
- **Из расплывчатой идеи надо родить настоящий многостраничный план** — **заспавнить писателя**, как со спекой, чтобы не размывать контракт. Граница: «переписать существующий список в чек-лист» = сам; «придумать структуру с нуля» = писатель.

В любой ветке итог один: в репозитории появляется чек-лист этапов. Закоммитить: `git add ROADMAP.md && git commit -m "chore: bootstrap roadmap"`.

### Выбор этапа

Выбрать **следующий незавершённый** этап (`- [ ]`). Подтвердить выбор с пользователем перед стартом.

Это же правило задаёт **resume** в новой сессии: открыть роадмап, найти первый незакрытый пункт — продолжить с него.

## Шаг 2. Конвейер этапа

Каждый этап проходит пять фаз в указанном порядке. Следующая фаза начинается только когда предыдущая прошла гейт ревью.

### Фаза A. Спека

1. Оркестратор ведёт диалог с пользователем: цели этапа, ограничения, решения по дизайну. Резюмировать решения списком.
2. Заспавнить агента-писателя:
   - Передать: резюме решений, этап из роадмапа, абсолютный путь к `superpowers:brainstorming` SKILL.md
   - Инструкция: прочитать SKILL.md brainstorming перед написанием, создать документ спеки в `docs/superpowers/specs/`
3. **Коммит**: `git add docs/superpowers/specs/ && git commit -m "spec: [название этапа]"`
4. **Гейт**: кросс-модельная волна ревью спеки (см. Шаг 3)
   - После каждого фиксера: `git add -A && git commit -m "fix(spec): [краткое описание правки]"`

### Фаза B. План

1. Заспавнить агента-писателя плана:
   - Передать: путь к спеке, этап из роадмапа, абсолютный путь к `superpowers:writing-plans` SKILL.md
   - Инструкция: прочитать SKILL.md writing-plans перед написанием, создать план в `docs/superpowers/plans/`
   - **Дополнительно** попросить выдать **манифест параллелизма** в конце плана: какие задачи можно делать одновременно (формат → `references/parallel-execution.md`)
2. **Коммит**: `git add docs/superpowers/plans/ && git commit -m "plan: [название этапа]"`
3. **Гейт**: кросс-модельная волна ревью плана (см. Шаг 3)
   - После каждого фиксера: `git add -A && git commit -m "fix(plan): [краткое описание правки]"`

### Фаза C. Реализация

Если в плане есть манифест параллелизма и в нём независимые задачи — реализация идёт **волнами параллельных агентов** (механика → `references/parallel-execution.md`). Иначе — последовательно, один реализатор за раз.

**Последовательный путь (нет манифеста / все задачи пересекаются):**
1. Заспавнить агента-реализатора:
   - Передать: путь к плану, абсолютный путь к `superpowers:subagent-driven-development` SKILL.md
   - Инструкция: прочитать SKILL.md перед реализацией
   - Выбрать модель по рубрике → `references/model-routing.md`

**Параллельный путь (есть независимые задачи):**
1. Построить волны по манифесту (готовые + непересекающиеся `touches`, кап ширины 3).
2. Каждого реализатора волны заспавнить в **изолированной копии** (`Agent` с `isolation: "worktree"`), с контрактом «git-слепого» агента (запрет на ветко-команды и выход за свою полосу).
3. Каждый агент коммитит у себя в worktree.
4. **Fan-in** (только оркестратор): проверить дифф каждого агента против заявленного `touches` → принять или отбросить+эскалировать → слить принятые по очереди → **коммит точки слияния**.
5. Пересчитать разблокированные задачи → следующая волна.

В обоих путях: внутренние per-task ревью superpowers **не считаются** волнами по R4 — они остаются внутри фазы C.

### Фаза D. Ревью кода

1. **Гейт**: кросс-модельная волна ревью кода (см. Шаг 3)
   - Диапазон ревью: весь дифф этапа (BASE..HEAD по всем задачам фазы C)
   - После каждого фиксера: `git add -A && git commit -m "fix(code): [краткое описание правки]"`

После прохода гейта D код этапа готов, но переходим к Фазе E — этап не закрывается, пока документация не приведена в соответствие.

### Фаза E. Документация

Код отгружен, но документация всё ещё описывает прошлое состояние. Эта фаза приводит её в соответствие с тем, что реально изменил этап. Обновляется **по диффу этапа** (что фактически поменялось), а не переписывается целиком.

**E1. Обнаружение поверхностей документации** (оркестратор сам — это учёт контекста, исключение 2).

Пройти по категориям, собрать список **существующих** в проекте:
- **Вики-папки**: `wiki/`, `docs/`, `documentation/`, `.wiki/`, GitHub Wiki
- **Контекст-папки**: `context/`, `контекст/`, `.context/`, `ai-context/`, `llm-context/`
- **MD рядом с кодом**: корневой `README.md` и per-package README, `ARCHITECTURE.md`, `CONTRIBUTING.md`, `CHANGELOG.md`, ADR (`docs/adr/`, `docs/decisions/`), doc-комментарии модулей
- **Доксайты / спеки API**: `mkdocs.yml`, конфиги Docusaurus/Mintlify, OpenAPI/Swagger, Storybook
- **Прочее**, что укажет пользователь

**Исключить из целей:** артефакты самого конвейера — `docs/superpowers/specs/` и `docs/superpowers/plans/`. Это источник истины этапа, он уже актуален; doc-фаза его не трогает.

Сопоставить дифф этапа с найденными поверхностями: новая фича → README + CHANGELOG; смена API → API-спека + вики; новое архитектурное решение → ADR/ARCHITECTURE. Цель — список конкретных файлов, которые этот этап обязан обновить.

Если поверхностей не нашлось — короткий диалог (исключение 1): спросить пользователя, где живёт документация и нужно ли её завести. Структуру с нуля не выдумывать молча; при согласии пользователя — это уже авторство, заспавнить писателя (как в ветке 2–3 Шага 1).

**E2. Спавн агента-писателя документации:**
- Передать: git-диапазон этапа (BASE..HEAD), путь к спеке, путь к плану, список целевых поверхностей с краткой ролью каждой
- Инструкция: обновлять **только** то, что изменил этот этап; зеркалить существующий стиль и структуру каждой поверхности (сами доки — образец формата); не переписывать целиком и не плодить новых разделов без необходимости
- Модель: ярус 2 по рубрике (`model-routing.md`) — интеграционная работа по готовому диффу; ярус 3, если нужен архитектурный нарратив (ADR, redesign-заметка)

**E3. Коммит**: `git add -A && git commit -m "docs: [название этапа]"`

**E4. Гейт**: кросс-модельная волна ревью документации (см. Шаг 3, промпт → `references/prompts/doc-review.md`)
- Фокус ревью: документация **точно отражает отгруженное поведение** (нет дрейфа док↔код), полнота (всё значимое из диффа отражено), валидность ссылок и примеров
- После каждого фиксера: `git add -A && git commit -m "fix(docs): [краткое описание правки]"`

После прохода гейта E этап завершён. Отметить в роадмапе (`- [x]`), закоммитить:
`git add ROADMAP.md && git commit -m "chore: mark [название этапа] complete"`

## Шаг 3. Цикл волны кросс-модельного ревью

Полная механика — в `references/review-protocol.md`.

Кратко:
1. Выбрать ревьюера: **другая модель/CLI, чем писатель**. Для Claude Code-оркестратора — лестница транспорта: `codex` → при исчерпании лимитов codex `opencode + deepseek-v4-pro` → честная деградация (матрица и рецепты → `references/spawn-recipes.md`)
2. Построить нейтральный промпт по шаблону из `references/prompts/` (spec/plan/code/doc)
3. Заспавнить ревьюера
4. Разобрать находки по контракту (Critical/Important/Minor)
5. Если Critical или Important есть:
   - Заспавнить фиксера с находками дословно
   - Счётчик волн < кап (по умолчанию 5)? → вернуться к п.2
   - Счётчик ≥ кап → эскалировать к пользователю
6. Если только Minor или пусто → **гейт пройден**

## Шаг 4. Завершение этапа

Отметка `- [ ]` → `- [x]` уже поставлена в конце Фазы E (после гейта документации). Здесь:
- Предложить пользователю перейти к следующему незавершённому этапу или остановиться
