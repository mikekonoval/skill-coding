# Model Routing

Рубрика выбора пишущей модели под задачу. Применяется оркестратором при спавне писателей и реализаторов.

**Принцип:** наименее мощная модель, которая справится с ролью.

---

## Три яруса способностей

### Ярус 1 — Механический (дешёвый/быстрый)

**Сигналы:**
- 1-2 файла, изолированная функция с полной спецификацией
- Переименование, рефакторинг без изменения логики
- Форматирование, миграция данных по шаблону
- Задача полностью описана планом — нет дизайн-суждений

**Когда НЕ применять:**
- Задача требует понимания контекста нескольких файлов
- Нет полного описания входов и выходов

### Ярус 2 — Интеграционный (стандартный)

**Сигналы:**
- 3-10 файлов, координация между модулями
- Соответствие паттернам существующего кода (нужно чтение кодобазы)
- Обработка ошибок, граничные случаи
- Дебаггинг с неизвестной причиной

### Ярус 3 — Архитектурный (верхний)

**Сигналы:**
- Авторство спеки или плана (любой этап)
- Финальное ревью кода (любой волны)
- Кросс-модульная архитектура, новые абстракции
- Задача с неоднозначным скоупом

**Правило неоднозначности:** если сложность задачи не очевидна → маршрутизировать на ярус выше, никогда вниз.

Оркестратор работает на **сильнейшей доступной модели** (сегодня Fable 5) — описывается как ярус, не хардкодится.

---

## Как определить ярус для задачи реализации

Читается из списка задач плана (Implementation Units):

1. Посчитать файлы в секции `Files:` — 1-2 → ярус 1, 3-10 → ярус 2, 10+ → ярус 3
2. Есть секция `Approach:` с дизайн-суждениями? → минимум ярус 2
3. Задача типа «авторство» (спека, план) или «ревью» → ярус 3

Для спека/плана — всегда ярус 3 независимо от числа файлов.

---

## Re-dispatch при BLOCKED

Если реализатор вернул статус `BLOCKED`:

1. Прочитать причину блока
2. Если причина — недостаток контекста: предоставить контекст и повторно заспавнить с той же моделью
3. Если причина — задача сложнее ожидаемого: повторно заспавнить с моделью яруса выше

Никогда не игнорировать `BLOCKED` и не повторять с той же моделью без изменений.

## Кросс-модельная матрица (кто кого ревьюит)

Правило: **ревьюер ≠ модель/CLI писателя**.

| Harness оркестратора | Писатель/реализатор | Ревьюер |
|---------------------|--------------------|----|
| Claude Code | Claude-агент (любой ярус) | `codex exec -s read-only` → при исчерпании лимитов codex: `opencode run --agent plan -m deepseek/deepseek-v4-pro` |
| Claude Code | `codex exec` | Claude-агент (ярус 3) |
| Codex (запасной) | Codex-агент | DeepSeek (opencode / deepcode CLI / API) |

Лестница ревьюера для Claude Code (codex → opencode+DeepSeek → честная деградация) — в `spawn-recipes.md`. DeepSeek ≠ Claude, поэтому fallback сохраняет кросс-модельность.

Если ревьюер недоступен — применить правило честной деградации (→ `spawn-recipes.md`), не подменять молча.
